\chapter{User's Guide}
\label{toc:userguide}

The \FramaC/\Region computes a single \emph{memory map} for every
\textsf{C}-function in the source. The computation of memory maps is performed
on-demand through the \FramaC GUI or through the \textsf{API} or once and for
all functions by using the \verb$-region$ command-line option.

The console output generated by the \verb$-region$ is test-oriented and might be
difficult to read by normal users. It consists of a text-based dump of all
regions for all functions with their attributes and properties.

Region maps are visual and interactive data that deserve a graphical user
interface to be exploited. Hence, the recommended way of using \FramaC/\Region
is \emph{via} the \FramaC GUI and its \textsf{Region Analysis} component:

\begin{logs}
> frama-c-gui -region *.c
\end{logs}

The Region Analysis view (See Figure~\ref{fig:ivette}) shows the \emph{memory
map} associated with the currently selected function. You can pin the current
map if you want to consult other functions while keeping the current map
visible. You can also download the current memory map in
DOT/Graphviz\footnote{\url{https://graphviz.org}} format.

\section{Walking through Region Maps}
\label{toc:navigation}

A \FramaC/\Region memory map consists of different kinds of cells or
\emph{regions}. Intuitively a \emph{region} is a delimited range of memory
locations with consistent access.

Regions can be split into \emph{subregions} to represent compound data
structures such as structures and arrays. Typically, \C variables of atomic type
are represented as a single memory region and \C arrays by a collection of
memory regions. However, pointer casts may also reshape objects into a different
layout. For instance, a 64-bits integer variable might be represented as an
\emph{array} of four 8-bits \emph{subregions}. Hence, in region maps there is no
difference between \C atomic types and \C arrays since both kind of \C objects
can be represented by: either a \emph{single} memory region; or an \emph{array}
of subregions depending on the actual memory access done by the code.

There is also a clear distinction between compound regions that are accessed
like \emph{arrays} of subregions and those that are accessed by individual
\emph{structure fields}. Notice that \emph{union fields} are not represented in
memory maps, although union fields actually reshape their contents and hence the
associated subregions.

\paragraph{Variables.}
The root regions of memory maps consists of the \C variables. Function returned
value has also its own region, denoted by \verb+\result+ like in \acsl. Some
regions can also be named with \emph{labels} from user's region annotations
(Cf.~\ref{toc:annotations}):

\paragraph{Atomic Regions.} The \emph{atomic} regions in memory maps consist
of aligned (arrays of) integer or float values. Inside an atomic region, all
values are accessed through a consistent unique \C type. There are three kinds
of access: read (R), written (W) and initialized (I):
\begin{quote}
\begin{tabular}{cl}

  \raisebox{-10pt}{\includegraphics[scale=0.6]{images/nodeR.pdf}}
  & Region with read-only access of type \verb+t+\\

  \raisebox{-10pt}{\includegraphics[scale=0.6]{images/nodeW.pdf}}
  & Region with write-only access of type \verb+t+\\

  \raisebox{-10pt}{\includegraphics[scale=0.6]{images/nodeRW.pdf}}
  & Region with read-write access of type \verb+t+\\

  \raisebox{-10pt}{\includegraphics[scale=0.6]{images/nodeRWI.pdf}}
  & Region with read-write access \emph{and} initialization\\

\end{tabular}
\end{quote}
Atomic types are indicated with a single-letter for concision but their complete
description (with size in bits) is accessible by hovering or clicking the
region. One-letter types are listed in Figure~\ref{fig:types}.

\paragraph{Pointer Regions.} When region values are pointers (\verb+*+), they
shall all consistently points to a \emph{unique} target memory region. This is
enforced during the construction of memory maps by merging and reshaping the
different pointed regions when necessary. When a region contains pointers that
points to another region, it is illustrated by a \emph{directed} edge in the
memory map:
\begin{quote}
\begin{tabular}{cl}

  \raisebox{-10pt}{\includegraphics[scale=0.6]{images/pointerR.pdf}}
  & Pointer region with its (constant) points-to target region\\

  \raisebox{-10pt}{\includegraphics[scale=0.6]{images/pointerRW.pdf}}
  & Pointer region with its (written) points-to target region\\

\end{tabular}
\end{quote}

\paragraph{Garbled Regions.} When regions with different types are merged, the
resulting region is said to be \emph{garbled}. Its is labeled with (\verb+x+)
one-letter type and its size is set to the greatest common divisor (in bits) of
the size of all accessed types, and their regions pointed-to, if any, are also
merged. Garbled regions are highlighted in red in memory regions:

\begin{quote}
\begin{tabular}{cl}

  \raisebox{-10pt}{\includegraphics[scale=0.6]{images/garbled.pdf}}
  & Garbled region, here with read-write access\\

\end{tabular}
\end{quote}

\paragraph{Compound Regions.} Contrary to arrays, structures and unions have
distinct memory layouts for each of their fields. Moreover, union fields
overlap with each others and padding bits might be inserted between structure
fields. Finally, not all fields of a given structure or union might be actually
accessed by the code.

When building memory maps, only \emph{structure} fields that are
\emph{explicitly} accessed by the code are represented. Union fields are
\emph{not} represented, although they reshape the layout of accessed values and
makes the associated subregions to be merged, which might in turn produce
garbled regions.

By construction, structure fields lead to \emph{separated} subregions of the
compound region, unless they are explicitly aliased by the code or user
annotations (Cf.~\ref{toc:annotations}). Hence, compound regions are
graphically represented by tables with disjoint rows that represent separated
subregions of each compound cell:
\begin{quote}
\begin{tabular}{cl}

  \raisebox{-25pt}{\includegraphics[scale=0.6]{images/compound.pdf}} &
  \parbox{10cm}{%
  Compound region with two distinct subregions for fields %
  \texttt{f} and \texttt{g} (other fields are not represented here)%
  }

\end{tabular}
\end{quote}
Like variables, compound \emph{array} fields are not explicitly represented,
although each field might contain either one single memory cell or an array of
\verb+N+ memory cells of some subregion. Padding bits and non-accessed fields
are represented by the number of bits they spawn over:
\begin{quote}
\begin{tabular}{ll}

  \raisebox{-8pt}{\includegraphics[scale=0.5]{images/fieldPAD.pdf}}
  & \texttt{N} bits of padding or non-accessed fields (no associated subregion) \\

  \raisebox{-8pt}{\includegraphics[scale=0.5]{images/fieldF.pdf}}
  & Subregion for field \texttt{f} \\

  \raisebox{-8pt}{\includegraphics[scale=0.5]{images/fieldFA.pdf}}
  & Array subregion of \texttt{N} memory cells for field \texttt{f} \\

\end{tabular}
\end{quote}
When consecutive fields are merged, the starting and final fields are indicated.
In some weird situation with garbled regions, some rows might not be kept
aligned with any structure fields, in which case they are represented by a
range of offsets (in bits) inside the memory region:
\begin{quote}

\section{Adding Region Annotations}
\label{toc:annotations}

By default, the \Region plug-in guesses an optimistic aliasing context
and assumes that pointers have all desirable good properties to avoid any
unexpected, unspecified or undefined behaviors. The optimistic guess covers the
following desirable properties:
\begin{itemize}
\item pointer variables only have either a null value, or an aligned
  address that fits within some object boundaries, up to the address immediately
  after (also called past-the-end of array elements);
\item formal parameters with pointer type initially point to valid, separated
  and properly initialized objects of the specified type;
\item pointed objects are considered to be writable unless their type have an
  explicit \verb+const+ attribute;
\end{itemize}
Would such an optimistic guess be incompatible with some code, the user shall
provide additional annotations for the \Region plug-in to be more
pessimistic and to consider more problematic cases to be properly handled and
analyzed.

\index{region@\verb+region+}
Region annotations are extended \acsl clauses, introduced by
\verb+region+ keyword (or \verb+\region::region+) that follow the syntactic
definitions of Figure~\ref{fig:annotations}. They can be used in function
contracts (default behavior only) or inside code-annotations.
Each region annotation introduces one or several regions specifications.
A typical example is:

\begin{lstlisting}
//@ region A: p->f[..], q[..], B: r[0..n], \nullable;
\end{lstlisting}

Such an annotation introduces two named regions, respectively \verb+A+ and
\verb+B+. In region named \verb+A+, we will find locations accessible through
\emph{paths} \verb+p->f[..]+ and \verb+q[..]+ ; hence for instance, l-values
\verb+p->f[4]+ and \verb+q[k]+ might be aliased. In region named \verb+B+, we
have a declared \emph{object} that might be available through \verb$r[k]$,
provided \verb$0<=k<=n$ (the specified range) and \verb$r!=\null$ (because of
the \verb+\nullable+ flag).

More generally, a region can be anonymous or identified with a \emph{name}.
Regions, with or without names, consists of \emph{paths} and \emph{objects}.
Paths are limited forms of \acsl terms denoting classes of l-values that might
be aliased with each others. They are roughly normal l-values with only
imprecise array access \verb$p[..]$, like in example region \verb+A+ above.
Objects are used to declare arrays thaht might be available at the beginning of
a function from its calling contexts, together with their range and specific
properties (see below). They are always in form $p[a..b]$ where $p$ is a single
l-value and $a$ and $b$ two integer values, and can be attributed with different
property flags, like in example region \verb+B+ above. Objects can only be
declared in function contracts.

We now describe region specifications, paths and objects in more details.

\paragraph{Region Names.} Inside a given function code or function
specification:
\begin{itemize}
  \item all regions with the same name are merged into the same region;
  \item two regions with different names may be merged if they share aliased path;
  \item two objects declared within regions with different names shall be
  separated from each other, even if they finally belong to the same (merged)
  region.
\end{itemize}

\paragraph{Aliasing Paths.}
Aliasing paths are regular \acsl term l-values, with the following
restrictions:
\begin{itemize}
\item no arithmetic conversion, only pointer casts are allowed;
\item array access shall have an unspecified range of indices \verb$a[..]$;
\item pointer access are considered just like undefined array access.
\end{itemize}
Should paths \verb+p+ and \verb+q+ are aliased with each other and belong the
same region $R$, then access paths \verb+*p+ and \verb+*q+ denote the
\emph{same} region $R'$ and region $R$ \emph{points-to} region $R'$.

\paragraph{Aliased Fields.}
Aliased fields specifications, for instance \verb+a.f..a.g+, must share the same
base aliasing path. Given any compound object accessible through path \verb+a+,
the entire space occupied between fields \verb+f+ and \verb+g+ will be merged
into the same region.

\paragraph{Objects.}
They shall be only specified in function contracts. An object \verb+a[p..q]+
consists of a term \verb+a+ of pointer type, and two integer terms \verb+p+ and
\verb+q+. The three terms \verb+a+, \verb+p+ and \verb+q+ are meant to be
evaluated in the pre-state of the function, just like pre-conditions.

An object specification \verb+p[a..b]+ denotes a \emph{potentially} existing
array of elements \verb$*(p+k)$ with index \verb+k+ ranging from \verb+p+ to
\verb+q+, both included.

Anonymous objects might overlap with each others unless they are declared in
different region clauses.

On the contrary, any two objects declared in regions with \emph{different} names
shall be \emph{separated} from each others. Sometimes, two objects with
different names will be later merged into the same region because of further
aliasing path specifications. However, provided they are introduced by region
with \emph{distinct} names, they shall be \emph{separated} at the function entry
point. Otherwise, objects declared within regions with the same name might
overlap with each others.

\paragraph{Object Properties.}
By default, any declared object \verb+p[a..b]+ is assumed to be valid and
properly initialized. Flags can be used in region specifications to downgrade
those hypotheses to unusual situations. The available flags for an
object specification \verb+a[p..q]+ are:
\index{nullable@\verb+\nullable+}
\index{allocated@\verb+\allocated+}
\index{garbage@\verb+\garbage+}
\index{readonly@\verb+\readonly+}
\begin{itemize}
  \item \verb+\nullable+ allows \verb+a+ to be \verb+\null+;
  \item \verb+\allocated+ allows range \verb$a+(p..q)$ to be valid partially
  valid or not valid at all;
  \item \verb+\garbage+ allows element \verb+a[k]+ to be partially initialized
  or not initialized at all;
  \item \verb+\readonly+ requires \verb+a[p..q]+ to have read-only access only.
\end{itemize}
Flags can be combined with each other and apply to all objects that appears in
the same (named or anonymous) region clause. If a region clause introduces
several region groups, flags only applies to objects of the same subgroup,
whatever their names. A warning is emitted when flags have no object to apply on.

\paragraph{Formals with Compound Type.}
In case a formal parameter of a function has a compound (structure or union)
type, its value will be required be fully initialized, although this is not
explicitly required by the C standard. To downgrade this assumption, the region
property \verb+\gargabe+ shall be put on the considered formal parameter.

\paragraph{Values of Pointer Type.}
In case a variable or an l-value has a pointer type, its value is assumed to be
aligned with the expected pointed type. This hypothesis can be leveraged by
using the kernel command-line option \verb+-no-warn-unaligned-pointer+.

\noindent Additionally, provided the kernel command-line option
\verb+-warn-invalid-pointer+ has been set, a variable with pointer type shall
only be assigned the \verb+\null+ value or a pointer to a valid object
(including the final edge of the object, as formally specified by
\verb+\object_pointer+ in \acsl).

\section{Generating Side Conditions}
\label{toc:commandline}

Region specifications are taken as \emph{additional requirements} to other
\acsl clauses of a function. As such, they are not necessarily verified
or preserved by any execution of the function and for any calling context.
Hence, region specifications necessarily induce some \emph{side conditions} to
be verified to be correct. The \Region plug-in provides two ways for
generating side-conditions: a programmatic \textsf{API} for other
\FramaC plug-ins to verify these side conditions on their own, and a
command-line option to generate regular \acsl clauses that enforce
side-conditions to be verified.

This behavior is similar to what provides the \rte plug-in
(\verb+-rte+). Actually, the side-conditions generated by \Region
generally subsume necessary side-conditions to prove the absence of some
categories of \emph{Runtime Errors}, so that \verb+-region+ shall be used
\emph{before} \rte to obtain more precise annotations.

The following command line options can be used to tune the generated side
conditions of \Region plug-in:

\begin{description}
\item[\texttt{-(no)-region}] generate side-conditions by inserting additional
  \acsl clauses in the code (not by default).
\item[\texttt{-region-(check|assert)}] generate side-conditions by using respectively
  \verb+check+ (the default) or \verb+assert+ clauses.
\item[\texttt{-(no)-region-logic}] generate side-conditions for \acsl
  terms and predicates throughout the specifications (not by default).
\item[\texttt{-(safe|unsafe)-arrays}] generate strict array bounds
  side-conditions whenever possible (safe, by default) or not (unsafe).
\item[\texttt{-(no)-warn-unaligned-pointer}] generate side-conditions when
  accessing memory through misaligned pointers, or when assigning a non-aligned
  pointer value to a pointer variable or l-value (set by default).
\item[\texttt{-(no)-warn-invalid-pointer}] generate side-conditions when invalid
  pointers are assigned to pointer variables or l-values (\emph{not} set by
  default).
\end{description}
